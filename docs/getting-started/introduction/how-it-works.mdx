# How It Works

![Evaluation Flow](docs/assets/evaluation-flow-animated.svg)

## The Pipeline

fasteval runs your test through five stages. You write a test, decorate it with metrics, call `fe.score()`, and we handle the rest.

![Evaluation Pipeline Steps](docs/assets/evaluation-pipeline-steps.svg)

Each stage does one thing. Your app logic stays separate from evaluation logic.

## Step-by-Step Walkthrough

### Step 1: Decorate

Apply metric decorators to define your quality criteria:

```python
import fasteval as fe

@fe.correctness(threshold=0.8)    # Must be factually accurate
@fe.toxicity(threshold=0.95)      # Must be safe for users
def test_customer_support():
    ...
```

Decorators register their metrics and thresholds to a test context. Stack as many as you want — the test must pass ALL of them.

### Step 2: Test

Execute your LLM, agent, or application inside the test function:

```python
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
def test_customer_support():
    # Your application logic runs here
    response = support_agent("How do I cancel my subscription?")
    ...
```

Your code runs exactly as it would in production. We don't modify or intercept anything — we just observe the output.

### Step 3: Score

Record the output for evaluation using `fe.score()`:

```python
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
def test_customer_support():
    response = support_agent("How do I cancel my subscription?")

    # Record what to evaluate
    fe.score(
        actual_output=response,
        expected_output="Go to Account Settings > Subscription > Cancel",
        input="How do I cancel my subscription?"
    )
```

Nothing gets evaluated yet. `fe.score()` just captures the actual output, expected output, and context — think of it as staging data for the evaluation phase.

### Step 4: Evaluate

After your test function returns, fasteval's pytest plugin kicks in:

```python
# This happens automatically after your test function returns

for metric in registered_metrics:
    score = await metric.evaluate(
        actual_output=response,
        expected_output=expected,
        input=query
    )
    # score: 0.0 - 1.0
```

Each metric builds an evaluation prompt and calls an LLM judge. The judge scores the output and explains why.

### Step 5: Result

Scores get compared against thresholds. Pass or fail.

```
$ pytest test_support.py -v

test_support.py::test_customer_support PASSED

Evaluation Results:
  correctness: 0.92 (threshold: 0.80) ✓
  toxicity: 0.98 (threshold: 0.95) ✓

  Aggregate Score: 0.95
```

If ALL metrics hit their thresholds, the test passes. If ANY metric falls short, the test fails — and you get detailed reasoning explaining what went wrong.

## Under the Hood

### Evaluation Prompt Construction

When you call `fe.score()` with a `correctness` metric, fasteval constructs a prompt like:

```
You are an evaluation judge assessing the correctness of an AI response.

**User Query**: How do I cancel my subscription?

**Expected Answer**: Go to Account Settings > Subscription > Cancel

**Actual Response**: To cancel your subscription, navigate to your Account
Settings, find the Subscription section, and click Cancel. You'll be asked
to confirm.

**Task**: Rate how correctly the actual response answers the query compared
to the expected answer. Consider semantic equivalence, not exact wording.

Provide:
1. A score from 0.0 to 1.0
2. Brief reasoning for your score
```

### Score Computation

The LLM judge returns structured output:

```json
{
  "score": 0.92,
  "reasoning": "The response correctly describes the cancellation process
               and matches the expected navigation path. It provides
               additional helpful context about confirmation without
               contradicting the expected answer."
}
```

### Threshold Comparison

```python
if score >= threshold:
    return TestResult.PASS
else:
    return TestResult.FAIL
```

## Metric Types

fasteval ships two kinds of metrics, and they work very differently.

### LLM-Based Metrics

These call an LLM judge for semantic evaluation. **`correctness`** checks factual accuracy against your expected output. **`relevance`** measures how well the response actually answers the query. **`toxicity`** flags unsafe or inappropriate content. **`hallucination`** catches fabricated information — stuff the model just made up. **`faithfulness`** checks whether the response stays grounded in the context you provided. And **`coherence`** looks at logical flow and internal consistency.

In practice, these take ~100-500ms because of the API call. But they understand paraphrasing, so you're not stuck doing brittle string matching. Every LLM-based metric returns both a score and reasoning, which is invaluable when debugging failures.

### Deterministic Metrics

These are instant — no API call, pure code. **`contains`** checks for substring presence. **`exact_match`** does strict string equality. **`regex`** handles pattern matching. **`json_schema`** validates JSON structure. **`word_count`** enforces length constraints.

They're either binary or threshold-based, and they're what you want for format validation. No ambiguity, no latency.

### Combining Metrics

Stack decorators to mix both types in a single test:

```python
@fe.correctness(threshold=0.8)     # LLM-based
@fe.toxicity(threshold=0.95)       # LLM-based
@fe.contains()                     # Deterministic
@fe.json_schema()                  # Deterministic
def test_comprehensive():
    response = my_agent(query)
    fe.score(response, expected, schema=my_schema)
```

All metrics run in parallel. The test passes only if every single one meets its threshold.

## Pytest Integration

fasteval plugs into pytest's lifecycle through a custom plugin. You don't need to learn a new test runner or change your workflow.

```
pytest discovers test_*.py
         │
         ▼
    ┌─────────────────┐
    │ Collection Hook │  ← fasteval identifies decorated tests
    └────────┬────────┘
             │
             ▼
    ┌─────────────────┐
    │  Test Execution │  ← Your test function runs
    └────────┬────────┘
             │
             ▼
    ┌─────────────────┐
    │   fe.score()    │  ← Captures evaluation data
    └────────┬────────┘
             │
             ▼
    ┌─────────────────┐
    │ Teardown Hook   │  ← fasteval triggers evaluation
    └────────┬────────┘
             │
             ▼
    ┌─────────────────┐
    │ Report Results  │  ← Scores displayed in output
    └─────────────────┘
```

### Benefits of Pytest Integration

- **Familiar workflow**: `pytest test_file.py -v`
- **IDE support**: Click to run tests in VS Code, PyCharm
- **Parallel execution**: `pytest -n auto` for concurrent evaluation
- **CI/CD ready**: Standard exit codes (0 = pass, 1 = fail)
- **Rich output**: Detailed failure messages with reasoning
