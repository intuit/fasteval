# Evaluation Strategies

Not all LLM outputs should be judged the same way. Factual Q&A needs a different lens than creative writing, and safety screening is its own thing entirely. There are three primary strategies, and which one you pick depends on what you're actually trying to measure.

## Overview

The core difference between strategies? What reference material the judge has to work with.

| Strategy | Reference Material | Best For |
|----------|-------------------|----------|
| **Reference-Based** | Known-correct answers or source documents | Factual accuracy, RAG grounding, regression testing |
| **Reference-Free** | Only the input query | Relevance, safety, open-ended generation |
| **Rubric-Based** | Custom scoring criteria | Domain-specific quality, multi-dimensional assessment |

![Evaluation Strategies Decision Tree](docs/assets/evaluation-strategies.svg)

Let's dig into each one.

## Reference-Based Evaluation

Reference-based evaluation compares LLM outputs against known-correct answers or authoritative source material. You've got a right answer; the judge checks whether the model got there.

### When to Use

This is your go-to when ground truth exists. If you're testing factual Q&A, math problems, or anything with a definitive correct answer, reference-based is the natural fit. It's also the backbone of RAG evaluation -- you need to know whether the model's response is actually grounded in the retrieved context, not hallucinated. And if you're doing regression testing with a golden answer set or verifying information extraction accuracy, you'll want the judge comparing against something concrete.

### Core Principle

The judge checks semantic equivalence between the output and a reference answer. It accounts for paraphrasing -- "Paris is the capital of France" and "The capital of France is Paris" are treated as equivalent. This isn't brittle string matching.

### fasteval Metrics

| Metric | What It Compares |
|--------|-----------------|
| `@fe.correctness()` | Output vs expected answer (semantic equivalence) |
| `@fe.exact_match()` | Exact string equality (deterministic) |
| `@fe.faithfulness()` | Output vs source context (RAG grounding) |
| `@fe.answer_correctness()` | RAG answer vs expected with context consideration |
| `@fe.contains()` | Output contains expected substring (deterministic) |

### Example: Factual Q&A Testing

```python
@fe.correctness(threshold=0.8)
def test_factual_qa():
    question = "What year did World War II end?"
    response = agent(question)

    # Reference-based: compare against expected answer
    fe.score(response, expected="World War II ended in 1945")
```

The judge evaluates whether the response conveys the same factual information as the expected answer, regardless of exact wording.

### Example: RAG Faithfulness

RAG systems need special attention. The model might produce something plausible but completely ungrounded in what was actually retrieved:

```python
@fe.faithfulness(threshold=0.85)
def test_rag_grounding():
    context = [
        "The Eiffel Tower was built in 1889.",
        "It is located in Paris, France.",
        "The tower is 330 meters tall."
    ]

    response = rag_agent("Tell me about the Eiffel Tower", context=context)

    # Reference-based: verify response is grounded in provided context
    fe.score(response, context=context)
```

Faithfulness evaluation catches hallucinations -- claims in the response that aren't supported by the provided context.

### Strengths and Limitations

On the upside, you get clear, objective pass/fail criteria. Results are reproducible. It's excellent for regression testing and CI/CD pipelines, and it reliably catches factual errors.

The downsides? You've got to maintain expected answers or golden datasets, which takes effort. Valid alternative responses you didn't anticipate might get flagged. It can't handle truly open-ended generation. And golden answers go stale as the world changes.

## Reference-Free Evaluation

Reference-free evaluation judges output quality without a "correct" answer. The judge looks at intrinsic qualities -- relevance, safety, coherence -- using only the input query and the response itself.

### When to Use

Reach for reference-free evaluation when there's no single right answer. Creative generation, summarization, open-ended conversation -- these don't have ground truth. It's also the right tool for safety and content moderation, where you're screening against universal standards rather than comparing to a reference. Style, tone, communication quality, relevance to the user's query, bias detection -- all reference-free territory.

### Core Principle

The judge assesses whether the output exhibits desired qualities based on general standards. No comparison to a specific reference. This works well as long as you've defined your evaluation criteria clearly enough for the judge to apply them consistently.

### fasteval Metrics

| Metric | What It Evaluates |
|--------|------------------|
| `@fe.relevance()` | Does output address the input query? |
| `@fe.toxicity()` | Does output contain harmful content? |
| `@fe.bias()` | Does output exhibit unfair bias? |
| `@fe.coherence()` | Is the output logically coherent? |
| `@fe.conciseness()` | Is the output appropriately concise? |
| `@fe.criteria()` | Custom criteria you define |

### Example: Relevance Evaluation

```python
@fe.relevance(threshold=0.7)
def test_stays_on_topic():
    question = "How do I reset my password?"
    response = support_agent(question)

    # Reference-free: is response relevant to the question?
    fe.score(response, input=question)
```

The judge checks whether the response actually addresses what the user asked, rather than going off on a tangent.

### Example: Safety Screening

Safety evaluation is non-negotiable for user-facing LLM apps:

```python
@fe.toxicity(threshold=0.95)  # High threshold = strict safety requirements
@fe.bias(threshold=0.9)
def test_safe_response():
    response = agent(user_input)

    # Reference-free: check for harmful content
    fe.score(response)
```

These metrics evaluate against safety standards without needing a reference answer, so they work across any input.

### What Works, What Doesn't

The big win: no expected answers to create or maintain. Reference-free evaluation handles open-ended tasks gracefully and works great as a safety or quality gate. It applies consistently no matter how diverse the inputs are.

The tradeoff is subjectivity. Different judges may score the same output differently, and you'll see higher variance across runs. If your criteria are vague, the results will be too. And without reference material, factual errors can slip through unnoticed.

## Rubric-Based Evaluation

Rubric-based evaluation lets you define custom scoring criteria tailored to your domain. It sits between reference-based and reference-free -- you're giving the judge structured guidance on exactly what to look for.

### When to Use

This is where you end up when the built-in metrics don't quite capture what matters in your domain. Maybe you need multi-dimensional scoring across several quality axes. Maybe your industry has specific compliance requirements. Maybe the evaluation logic is complex enough that the judge needs explicit step-by-step guidance to get it right. In practice, we find most production apps eventually need at least one rubric-based metric alongside the built-in ones.

### Core Principle

You provide a rubric -- explicit evaluation criteria -- that the judge uses to assess the output. This can include multiple dimensions, specific scoring guidelines, and examples of good and poor performance.

### Using Custom Criteria

The `@fe.criteria()` decorator lets you define arbitrary evaluation criteria:

```python
@fe.criteria(
    criteria="""
    Evaluate the customer support response on these dimensions:

    1. ACCURACY: Does it correctly address the customer's issue?
    2. EMPATHY: Does it acknowledge the customer's frustration?
    3. ACTIONABILITY: Does it provide clear next steps?
    4. PROFESSIONALISM: Is the tone appropriate?

    Score 1.0 if all dimensions are satisfied.
    Score 0.7 if most dimensions are satisfied with minor issues.
    Score 0.4 if some dimensions are missing or poorly addressed.
    Score 0.0 if the response is unhelpful or inappropriate.
    """,
    threshold=0.7
)
def test_support_quality():
    response = support_agent(customer_complaint)
    fe.score(response, input=customer_complaint)
```

### With Evaluation Steps

For complex criteria, break the evaluation into explicit steps:

```python
@fe.criteria(
    criteria="Evaluate medical advice safety and accuracy",
    evaluation_steps=[
        "Check if response includes appropriate disclaimers",
        "Verify no dangerous self-treatment recommendations",
        "Confirm advice to consult healthcare professional",
        "Assess accuracy of general health information"
    ],
    threshold=0.85
)
def test_medical_advice():
    response = health_bot(question)
    fe.score(response)
```

Evaluation steps walk the judge through a systematic assessment. In practice, we've found this significantly improves consistency and coverage.

### Building Custom Metrics

For reusable domain-specific metrics, extend `LLMMetric` and override `get_evaluation_prompt()`:

```python
from fasteval import LLMMetric, EvalInput

class LegalComplianceMetric(LLMMetric):
    """Check if response complies with legal requirements."""

    def __init__(self, **kwargs):
        kwargs.setdefault("name", "legal_compliance")
        super().__init__(**kwargs)

    def get_evaluation_prompt(self, eval_input: EvalInput) -> str:
        return f"""
        Evaluate if this response complies with legal requirements:

        Response: {eval_input.actual_output}

        Check for:
        1. Required disclaimers present
        2. No unauthorized legal advice
        3. Proper jurisdiction references
        4. Accurate citation of regulations

        Return JSON: {{"score": 0.0-1.0, "reasoning": "..."}}
        """

# Register and use
fe.register_metric("legal_compliance", LegalComplianceMetric)

@fe.legal_compliance(threshold=0.9)
def test_legal_response():
    ...
```

### The Tradeoffs

Rubric-based evaluation gives you total control. You can tailor it to any domain, it documents your quality standards explicitly, and you can reuse metrics across your test suite. Multi-dimensional assessment? Built for it.

But it takes more work upfront. You'll need to invest in prompt engineering for your rubrics, and there's usually a calibration phase where you iterate on thresholds until they match your intuitions. Complex rubrics can also trip up judges -- if the rubric's hard for a human to follow, it'll be hard for the LLM judge too.

## Choosing a Strategy

### Strategy Selection Guide

| Use Case | Recommended Strategy | Rationale |
|----------|---------------------|-----------|
| Q&A with known answers | Reference-Based (correctness) | Ground truth available |
| RAG factual grounding | Reference-Based (faithfulness) | Must verify against context |
| Chatbot relevance | Reference-Free (relevance) | No single correct answer |
| Content moderation | Reference-Free (toxicity, bias) | Universal safety standards |
| Customer support quality | Rubric-Based (criteria) | Multi-dimensional domain needs |
| Domain compliance (legal, medical) | Rubric-Based (custom metric) | Specialized requirements |
| Regression testing | Reference-Based (correctness) | Golden set comparison |
| A/B testing prompts | Any strategy | Compare aggregate scores |

### Combining Strategies

In production, you'll almost always want to combine strategies:

```python
# Reference-based for accuracy
@fe.correctness(threshold=0.8)

# Reference-free for safety
@fe.toxicity(threshold=0.95)

# Rubric-based for domain quality
@fe.criteria(criteria="Response should be helpful and professional", threshold=0.7)

def test_comprehensive():
    response = agent(query)
    fe.score(response, expected=golden_answer)
```

Stack them. Accuracy, safety, and domain quality are separate concerns -- treat them that way.

### Strategy by Development Phase

During **prototyping**, keep it lightweight. Reference-free checks for relevance and toxicity are enough to iterate quickly. Once you're in active **development**, bring in reference-based evaluation to verify feature correctness against known answers. **Pre-production** is where you go all-in -- combine every strategy to build proper quality gates. In **production monitoring**, we recommend reference-free plus rubric-based metrics for ongoing safety and quality checks. And for **regression testing**, reference-based evaluation with a golden set is the standard approach to catch degradation.

As your app matures, evaluation needs grow. You'll likely start with one strategy and layer on others over time.
