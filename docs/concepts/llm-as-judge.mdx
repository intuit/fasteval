# LLM as a Judge

LLM-as-Judge is simple: you use one LLM to grade the output of another. Instead of writing fragile regex checks or manually reviewing thousands of responses, you let a capable model do the evaluating for you.

## What is LLM-as-Judge?

You define criteria. A "judge" model scores outputs against those criteria and explains its reasoning. Feed it the output, some reference material, and scoring instructions — it hands back a score and a rationale.

![LLM as a Judge Flow](docs/assets/llm-as-judge-flow.svg)

**Example in fasteval:**

```python
import fasteval as fe

@fe.correctness(threshold=0.8)  # Judge evaluates correctness
def test_qa_response():
    response = my_agent("What is the capital of France?")
    fe.score(response, expected="Paris is the capital of France")
```

Behind the scenes, fasteval constructs a prompt asking the judge LLM to evaluate whether the response is semantically correct compared to the expected answer.

## Research Foundation

There's a growing body of research backing this pattern up. Here's what we think matters most.

### It actually agrees with humans

The MT-Bench study (Zheng et al., 2023) found GPT-4 agreed with human annotators over 80% of the time on pairwise comparisons. AlpacaEval showed similar results — LLM judges rank outputs about as consistently as human annotators agree with each other. And JudgeLM (Zhu et al., 2023) pushed it further by fine-tuning judge models for even higher correlation. So the agreement is real, not just anecdotal.

### Position bias is a thing

LLM judges sometimes prefer whichever response appears first (or second) in the prompt, regardless of quality. The fix is straightforward: randomize response order, run multiple evaluations and average, and use structured output formats that reduce ordering effects.

### Self-enhancement bias

Judges tend to favor outputs that sound like something they'd generate. In practice, we've found a few things help: use a different model family for judging than you used for generation, calibrate thresholds with known-good and known-bad examples, and mix in deterministic checks where you can.

## How It Works

The judge LLM receives a structured prompt containing:

1. **The output to evaluate** — What your LLM generated
2. **Reference/criteria** — Expected answer, context, or evaluation rubric
3. **Scoring instructions** — How to assign a score (typically 0.0 to 1.0)

The judge returns a numerical score plus reasoning that explains the assessment. That reasoning matters a lot for debugging — when a test fails, the "why" is right there.

### Evaluation Prompt Structure

fasteval builds evaluation prompts following a few key principles. The judge gets a clear role definition ("You are an expert evaluator...") along with explicit scoring criteria and examples of each score level. Outputs come back as structured JSON so parsing is reliable. And the evaluation task is kept separate from the content being evaluated, so the judge doesn't get confused about what it's actually scoring.

## When to Use LLM-as-Judge

LLM-as-Judge shines when you need semantic understanding. It handles **semantic correctness** well because it gets paraphrasing, synonyms, and equivalent expressions. It's great at judging **relevance to a question** — understanding context and intent, not just keyword overlap. **Tone and style** assessments come naturally to it. And it can evaluate **safety and toxicity** since these models are trained to spot harmful content. It's also solid at **factual grounding** when you provide reference context to check against.

But don't use it for everything. If you need exact string matching, use `@fe.exact_match()`. JSON schema validation? That's `@fe.json_schema()`. Keyword presence is just `@fe.contains()`. Numerical calculations and format compliance belong to deterministic checks — regex, schema validators, plain old assertions. Cheaper, faster, and more reliable for those jobs.

## Strengths

### Semantic Understanding

Unlike exact string matching, LLM judges understand that these expressions convey the same meaning:

- "Paris is the capital"
- "The capital is Paris"
- "France's capital city is Paris"

This lets evaluation focus on meaning rather than surface form. That's the whole point when you're evaluating natural language.

```python
@fe.correctness(threshold=0.8)
def test_semantic_equivalence():
    response = agent("Capital of France?")
    # All of these would score high:
    fe.score("Paris is France's capital city", expected="The capital of France is Paris")
```

### Nuanced Assessment

LLM judges can assess subtle qualities that rule-based systems can't touch. Is the response helpful but not rambling? Does it acknowledge uncertainty when it should? Is the tone right, professional but not stiff?

```python
@fe.criteria(
    criteria="Response should be helpful, concise, and acknowledge when unsure",
    threshold=0.7
)
def test_response_quality():
    response = agent("What will Bitcoin be worth next year?")
    fe.score(response)  # Judge evaluates against the criteria
```

### Scalability

You can run thousands of evaluations without a single human reviewer. The same criteria get applied every time, so scoring stays consistent. It plugs straight into CI/CD pipelines for continuous quality monitoring. And at scale, it's dramatically cheaper than paying people to review outputs.

## Limitations and Mitigations

### Judge Bias

The judge LLM has its own biases baked in from training. It might prefer verbose responses over concise ones, or get fooled by confident-sounding answers that are flat wrong. Different models will also score the same output differently.

**Mitigations:**

- Use consistent judge models across evaluations
- Calibrate thresholds empirically with known examples
- Consider multiple judge models and averaging for critical decisions
- Set temperature to 0 for more deterministic evaluations

### Cost Considerations

Each evaluation requires an LLM API call:

| Scenario | Calculation | Cost (GPT-4o-mini) | Cost (GPT-4o) |
|----------|-------------|--------------------|-----------------------|
| 100 tests, 5 metrics each | 500 calls | ~$0.04 | ~$0.63 |
| 1,000 tests, 3 metrics each | 3,000 calls | ~$0.23 | ~$3.75 |
| Daily CI run | Variable | Budget accordingly | Reserve for critical paths |

**Mitigations:**

- Use cheaper models (GPT-4o-mini, Claude Haiku) for most tests
- Reserve expensive models for critical evaluation paths
- Combine with deterministic checks to reduce LLM calls
- Use sampling strategies for high-volume production evaluation

### Non-Determinism

The same input may receive slightly different scores across runs. That's just how LLMs work.

**Mitigations:**

- Set `temperature=0` for the judge model
- Use consistent prompts and avoid ambiguous criteria
- Build in threshold margins (if scores cluster around 0.85, don't set your threshold at 0.85)
- For critical decisions, run multiple evaluations and average

## LLM-as-Judge vs Human Evaluation

| Aspect | LLM-as-Judge | Human Evaluation |
|--------|--------------|------------------|
| **Speed** | Milliseconds per evaluation | Minutes to hours |
| **Cost at scale** | ~$0.001/evaluation | ~$1-10/evaluation |
| **Consistency** | High with same prompt | Variable (inter-rater disagreement) |
| **Nuance** | Good for defined criteria | Excellent for edge cases |
| **Domain expertise** | Limited to training data | Can leverage deep domain knowledge |
| **Automation** | Fully automated | Requires human availability |
| **Explainability** | Provides reasoning | Can explain complex judgments |

**When to use humans:**

- Validating your LLM-as-Judge setup (meta-evaluation)
- Handling edge cases and ambiguous scenarios
- High-stakes decisions requiring accountability
- Building calibration datasets for threshold tuning

**Hybrid approach:**

What we've found works best is combining both. Use LLM-as-Judge for the high-volume automated runs, then have humans review a sample to validate and calibrate. You get coverage and accuracy without burning your budget.

```python
@fe.correctness(threshold=0.8)
@fe.human_review(sample_rate=0.1)  # Human reviews 10% of cases
def test_support_response():
    response = agent(query)
    fe.score(response, expected_response)
```

## Best Practices

### 1. Choose the Right Judge Model

Model selection affects both evaluation quality and cost:

```python
# Fast and cheap for most tests
fe.set_default_provider(OpenAIClient(model="gpt-4o-mini"))

# More capable for critical evaluations
@fe.correctness(model="gpt-4o")
def test_critical_path():
    ...
```

**Model selection guidelines:**
- **GPT-4o-mini / Claude Haiku**: Development, CI/CD, high-volume testing
- **GPT-4o / Claude Sonnet**: Production quality gates, nuanced evaluation
- **GPT-4-turbo / Claude Opus**: Complex reasoning, multi-step evaluation

### 2. Set Temperature to Zero

For reproducible evaluations:

```python
client = OpenAIClient(model="gpt-4o-mini", temperature=0)
fe.set_default_provider(client)
```

### 3. Calibrate Thresholds Empirically

Don't guess thresholds — measure them with known examples:

1. Collect a set of known-good and known-bad responses
2. Run evaluation on all examples without thresholds
3. Analyze the score distribution
4. Set threshold to separate good from bad with appropriate margin

### 4. Combine with Deterministic Checks

Use LLM-as-Judge for semantic qualities, deterministic checks for structure:

```python
@fe.correctness(threshold=0.8)      # Semantic: Is it right?
@fe.json_schema(schema=my_schema)   # Deterministic: Valid JSON?
@fe.contains(expected="disclaimer") # Deterministic: Has disclaimer?
def test_api_response():
    response = agent(query)
    fe.score(response, expected)
```

### 5. Review Reasoning on Failures

When a test fails, always check the judge's reasoning:

```python
@fe.correctness(threshold=0.8)
def test_with_review():
    response = agent(query)
    result = fe.score(response, expected)

    # Access reasoning for debugging
    for metric in result.metric_results:
        if not metric.passed:
            print(f"{metric.metric_name}: {metric.score}")
            print(f"Reasoning: {metric.reasoning}")
```

The reasoning helps you understand whether the issue is with your agent, your expected output, or your evaluation criteria.
