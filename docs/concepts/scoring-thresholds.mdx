# Scoring & Thresholds

If your thresholds aren't calibrated, you'll end up with one of two problems: flaky tests that fail randomly, or tests so permissive they let real issues through. This page covers how scores work and how to pick thresholds that do their job.

## Understanding Scores

All fasteval metrics return a score between **0.0 and 1.0**. It's a continuous confidence scale — how well does the output satisfy the evaluation criteria?

![Scoring Thresholds](docs/assets/scoring-thresholds.svg)

### Score Semantics by Metric

Different metrics interpret the 0.0-1.0 scale according to their evaluation criteria:

| Metric | 0.0-0.3 (Low) | 0.4-0.6 (Mid) | 0.7-1.0 (High) |
|--------|---------------|---------------|----------------|
| **correctness** | Factually wrong | Partially correct | Semantically correct |
| **relevance** | Off-topic, unrelated | Somewhat related | Directly addresses input |
| **toxicity** | Highly toxic content | Some concerns present | Safe and appropriate |
| **faithfulness** | Hallucinated content | Partially grounded | Fully grounded in context |
| **bias** | Clear unfair bias | Some bias detected | Fair and balanced |
| **coherence** | Incoherent, contradictory | Some logical issues | Logically consistent |

### Interpreting Score Results

fasteval gives you detailed score information, including reasoning:

```python
@fe.correctness(threshold=0.8)
def test_example():
    response = agent(query)
    result = fe.score(response, expected)

    # Access detailed results
    print(f"Aggregate Score: {result.aggregate_score}")

    for mr in result.metric_results:
        print(f"{mr.metric_name}: {mr.score}")
        print(f"  Threshold: {mr.threshold}")
        print(f"  Passed: {mr.passed}")
        print(f"  Reasoning: {mr.reasoning}")
```

Don't sleep on the `reasoning` field. It tells you *why* the judge gave that score, which makes the difference between staring at a failed test and actually knowing what to fix.

### Score Confidence Interpretation

Scores above 0.95 mean near-perfect output — the response fully satisfies the criteria. In the 0.85-0.94 range, you're looking at good output with minor issues, still generally acceptable. Drop to 0.70-0.84 and you've met basic requirements but nothing more. Once you're in the 0.50-0.69 zone, there are significant issues and the output probably isn't acceptable. Below 0.50 things are clearly broken — 0.30-0.49 means substantial problems, and anything under 0.30 is a straight-up failure.

## Score Distribution Patterns

In practice, LLM evaluation scores don't spread out evenly. They cluster. Understanding where they cluster makes threshold selection way easier.

### Typical Distribution

Good and bad outputs tend to land at opposite ends of the scale:

- **Good responses**: Typically cluster between 0.80-0.95
- **Bad responses**: Typically cluster between 0.20-0.50
- **Gap in the middle**: Makes threshold selection more straightforward

What we've found is that this bimodal pattern shows up because semantic correctness is pretty binary in practice — the output is either substantially right or it isn't. Minor phrasing differences don't move the needle nearly as much as fundamental correctness does.

### Why This Matters

The gap between good and bad score clusters determines how reliable your thresholds can be:

- **Large gap (>0.2)**: Easy to set thresholds that reliably separate good from bad
- **Small gap (<0.1)**: Thresholds may be fragile; consider improving evaluation criteria or using more capable judge models
- **Overlapping distributions**: Indicates ambiguous evaluation criteria; refine your prompts or split into multiple metrics

## Setting Thresholds

The threshold determines pass/fail. Score meets or exceeds it? Pass. Doesn't? Fail.

```python
@fe.correctness(threshold=0.8)  # Pass if score >= 0.8
def test_qa():
    ...
```

### Threshold Guidelines by Use Case

| Use Case | Recommended Threshold | Rationale |
|----------|----------------------|-----------|
| **Safety-critical (toxicity, bias)** | 0.95+ | Very low tolerance for harmful content |
| **Factual accuracy (correctness)** | 0.80 | Must be substantially correct |
| **RAG grounding (faithfulness)** | 0.85 | Strong fidelity to source material |
| **General quality (relevance)** | 0.70 | Some flexibility acceptable |
| **Creative tasks (criteria)** | 0.60 | Subjective, more lenient |
| **Regression testing** | 0.85 | Catch quality degradation |
| **Smoke tests** | 0.70 | Quick sanity check |

### Context-Dependent Thresholds

Different parts of your system deserve different threshold levels:

```python
# Strict for production-critical paths
@fe.correctness(threshold=0.9)
def test_payment_processing():
    ...

# Lenient for experimental features
@fe.correctness(threshold=0.7)
def test_new_feature_prototype():
    ...

# Very strict for safety
@fe.toxicity(threshold=0.98)
def test_user_facing_response():
    ...
```

## Calibrating Thresholds

Don't guess your thresholds. Measure them. The goal is finding values that cleanly separate acceptable outputs from unacceptable ones.

### Step 1: Collect Calibration Examples

Create a representative set of known-good and known-bad responses:

```python
CALIBRATION_SET = [
    # (response, is_good, rationale)
    ("Paris is the capital of France.", True, "Correct answer"),
    ("The capital of France is Paris, located on the Seine.", True, "Correct with context"),
    ("France is a country in Europe.", False, "Doesn't answer the question"),
    ("I don't know.", False, "Non-answer"),
    ("The capital is Lyon.", False, "Incorrect answer"),
]
```

Include edge cases and borderline examples — those are what'll actually stress-test your threshold.

### Step 2: Score All Examples

Run evaluation without thresholds to collect raw scores:

```python
import fasteval as fe

def calibrate_threshold():
    good_scores = []
    bad_scores = []

    for response, is_good, rationale in CALIBRATION_SET:
        # Evaluate without threshold to get raw score
        result = evaluate_without_threshold(response, "Paris is the capital of France")

        if is_good:
            good_scores.append(result.score)
        else:
            bad_scores.append(result.score)

    print(f"Good responses: min={min(good_scores):.2f}, max={max(good_scores):.2f}, avg={sum(good_scores)/len(good_scores):.2f}")
    print(f"Bad responses: min={min(bad_scores):.2f}, max={max(bad_scores):.2f}, avg={sum(bad_scores)/len(bad_scores):.2f}")

    # Calculate separation
    gap = min(good_scores) - max(bad_scores)
    print(f"Separation gap: {gap:.2f}")

    # Suggest threshold
    if gap > 0:
        suggested = (min(good_scores) + max(bad_scores)) / 2
        print(f"Suggested threshold: {suggested:.2f}")
    else:
        print("WARNING: Score distributions overlap - refine evaluation criteria")
```

### Step 3: Select Optimal Threshold

Pick a threshold that:

1. **Falls within the gap** between good and bad score distributions
2. **Includes a safety margin** to account for score variance
3. **Aligns with your risk tolerance** (stricter = fewer false passes, more false failures)

Example calibration result:

| Good Responses | Scores |
|----------------|--------|
| Correct answer | 0.92 |
| Correct with context | 0.88 |

| Bad Responses | Scores |
|---------------|--------|
| Doesn't answer | 0.45 |
| Non-answer | 0.38 |
| Incorrect | 0.32 |

- Min good score: 0.88
- Max bad score: 0.45
- Gap: 0.43 (excellent separation)
- Suggested threshold: 0.66 (midpoint)
- With safety margin: 0.75

### When Thresholds Don't Separate Well

If good and bad scores overlap significantly, you've got options:

1. **Improve the evaluation prompt**: Add more specific criteria or examples
2. **Use a more capable judge model**: GPT-4o vs GPT-4o-mini often improves separation
3. **Add reference context**: More information helps the judge make accurate assessments
4. **Split into multiple metrics**: Evaluate different aspects separately rather than combining

## Threshold Strategies

### Conservative Strategy (Minimize False Positives)

Use strict thresholds to catch more issues, accepting that some good responses may fail:

```python
@fe.correctness(threshold=0.9)   # High bar for accuracy
@fe.toxicity(threshold=0.98)     # Very strict on safety
```

**Appropriate when:**
- Safety-critical applications
- High cost of releasing low-quality outputs
- Regulatory compliance requirements
- Production quality gates

### Balanced Strategy (Typical Production)

Trade-off between catching issues and test stability:

```python
@fe.correctness(threshold=0.8)
@fe.relevance(threshold=0.7)
@fe.toxicity(threshold=0.95)
```

**Appropriate when:**
- Standard production testing
- CI/CD integration
- General quality assurance

### Lenient Strategy (Minimize False Negatives)

Lower thresholds for exploratory testing or when false alarms are expensive:

```python
@fe.correctness(threshold=0.6)   # Allow more variation
@fe.criteria(threshold=0.5)      # Subjective/creative tasks
```

**Appropriate when:**
- Prototyping and exploration
- Creative generation tasks
- Early development stages
- Smoke testing

### Environment-Specific Thresholds

Adjust thresholds based on environment:

```python
import os

# Stricter in CI, more lenient locally
THRESHOLD = 0.85 if os.getenv("CI") else 0.75

@fe.correctness(threshold=THRESHOLD)
def test_qa():
    ...
```

This lets you iterate fast locally while keeping strict quality gates in CI/CD.

## Common Pitfalls

### Threshold Too High

Tests fail even for obviously good responses. You'll see something like this:

```python
# Problematic: 0.95 is very strict
@fe.correctness(threshold=0.95)
def test_qa():
    response = "The capital of France is Paris."  # Score: 0.88
    fe.score(response, expected="Paris")  # FAILS despite being correct!
```

This usually happens when thresholds are set without calibration. Judge models tend to score conservatively — even clearly correct answers rarely hit 0.95. Paraphrased responses score a bit lower too. The fix: calibrate with real data, review why correct responses score lower than you'd expect, and ask whether your criteria are too strict.

### Threshold Too Low

The opposite problem — bad responses sneak through:

```python
# Problematic: 0.5 is too lenient
@fe.correctness(threshold=0.5)
def test_qa():
    response = "France is in Europe."  # Score: 0.55 (wrong answer!)
    fe.score(response, expected="Paris")  # PASSES despite being wrong!
```

We see this a lot when teams try to "fix" flaky tests by dropping their thresholds. That's treating the symptom, not the cause. Instead, calibrate against known-bad examples and fix the underlying flakiness directly.

### Not Accounting for Variance

LLM judges have inherent score variance. You need a margin. If good responses typically score 0.82-0.90, don't set your threshold at 0.82 — that's the floor. Use 0.78-0.80 instead:

```python
# If good responses typically score 0.82-0.90:
# - Don't set threshold at 0.82 (minimum good score)
# - Use 0.78-0.80 to account for variance

@fe.correctness(threshold=0.78)
```

### Same Threshold for All Metrics

Different metrics have different score distributions. Don't treat them the same:

```python
# Bad: same threshold for very different metrics
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.8)  # Toxicity should be stricter!

# Better: metric-appropriate thresholds
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
```

Safety metrics need higher thresholds than accuracy metrics. The cost of letting harmful content through is just way higher than the cost of a slightly wrong answer.

### Ignoring the Reasoning

When a test fails, always check the judge's reasoning:

```python
result = fe.score(response, expected)

if not result.passed:
    for mr in result.metric_results:
        if not mr.passed:
            print(f"Failed: {mr.metric_name}")
            print(f"Score: {mr.score}, Threshold: {mr.threshold}")
            print(f"Why: {mr.reasoning}")  # This explains the score!
```

The reasoning often reveals whether the issue is with:
- Your agent (needs improvement)
- Your expected output (too specific or incorrect)
- Your evaluation criteria (needs refinement)
