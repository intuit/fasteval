# Hallucination Metric

LLMs will confidently make things up. The hallucination metric catches fabricated or unsupported claims by checking whether statements in a response are actually grounded in the provided context or established facts.

## Overview

Hallucination is one of the hardest problems in LLM deployment. Models generate convincing but entirely fabricated information — fictional citations, invented statistics, made-up product features. If your app needs factual accuracy, you need to test for this.

### The Hallucination Problem

| Hallucination Type | Description | Example |
|-------------------|-------------|---------|
| **Factual errors** | Incorrect facts presented confidently | "Python was created in 1985" (actually 1991) |
| **Fabricated details** | Made-up specifics | Inventing citation sources |
| **Extrapolation** | Going beyond provided context | Adding unstated product features |
| **Conflation** | Mixing information incorrectly | Combining details from different sources |

### Why It Matters

Users can't tell the difference between hallucinated content and accurate information. That's the core problem. In legal, medical, or financial contexts, wrong information creates real liability. And once users catch your app making things up, trust is gone — they'll second-guess every response after that.

## Understanding Hallucination

### Score Interpretation

The metric evaluates every factual claim in the response against available context:

| Score | Interpretation | Typical Scenario |
|-------|---------------|------------------|
| **0.9-1.0** | Fully grounded | All claims supported by context or established facts |
| **0.7-0.9** | Minor unsupported details | Small embellishments that don't contradict context |
| **0.5-0.7** | Some fabrication | Mix of grounded and unsupported claims |
| **0.3-0.5** | Significant hallucination | Major unsupported or contradictory claims |
| **0.0-0.3** | Extensive fabrication | Little connection to provided context |

### What Gets Evaluated

The metric extracts factual claims from the response, then checks each one against the provided context. For claims that aren't in the context, it assesses general factual accuracy. It also specifically looks for contradictions — statements that directly conflict with what the context says.

## Parameters

```python
@fe.hallucination(
    threshold=0.85,   # Minimum score to pass (higher = fewer hallucinations)
    model="gpt-4o",   # Override the default LLM model
)
```

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `threshold` | float | 0.8 | Minimum score to pass (higher = stricter) |
| `model` | str | None | Override the default LLM model for this metric |

## Practical Examples

### RAG System Testing

For retrieval-augmented generation, verify responses stay grounded in retrieved documents:

```python
import fasteval as fe

@fe.hallucination(threshold=0.85)
def test_rag_grounding():
    docs = retriever.search("refund policy")
    context = "\n".join([doc.text for doc in docs])

    response = my_rag_agent("What's your refund policy?", context=context)
    fe.score(response, context=context)
```

### Knowledge Base Q&A

Verify the LLM doesn't invent information beyond what's in your knowledge base:

```python
@fe.hallucination(threshold=0.9)
def test_kb_accuracy():
    kb_entry = """
    Product: Widget Pro
    Price: $99
    Features: Wireless, Waterproof, 2-year warranty
    """

    response = my_llm("Describe Widget Pro", context=kb_entry)
    fe.score(response, context=kb_entry)
```

If the LLM says "Widget Pro has a 3-year warranty" when the context says 2 years, the score drops significantly.

### Combining with Correctness

Use both metrics for thorough accuracy evaluation:

```python
@fe.hallucination(threshold=0.85)  # Grounded in context
@fe.correctness(threshold=0.8)     # Semantically correct
def test_accurate_and_grounded():
    context = "The Eiffel Tower is 330 meters tall and was completed in 1889."

    response = my_llm("How tall is the Eiffel Tower?", context=context)
    fe.score(
        actual_output=response,
        expected_output="330 meters",
        context=context
    )
```

### High-Stakes Domains

For medical, legal, or financial content, use strict thresholds:

```python
@fe.hallucination(threshold=0.95)  # Very strict
@fe.contains()  # Required disclaimers
def test_medical_information():
    medical_context = load_verified_medical_content()

    response = health_bot("What are the symptoms of diabetes?", context=medical_context)
    fe.score(response, context=medical_context)
    fe.score(response, "consult a healthcare professional")  # Disclaimer check
```

## Hallucination vs Faithfulness

Both metrics detect fabricated information, but they're optimized for different scenarios:

| Aspect | Hallucination | Faithfulness |
|--------|--------------|--------------|
| **Primary use** | General fact-checking | RAG-specific evaluation |
| **Context handling** | Optional context | Requires retrieval context |
| **Focus** | All factual claims | Grounding in retrieved docs |
| **Best for** | Knowledge Q&A, general chatbots | RAG pipelines |

Use hallucination when you're testing general knowledge accuracy, when context is optional, or when evaluating standalone factual claims. Use faithfulness when you're building RAG systems with multiple retrieved documents where strict adherence to source material matters. For production RAG systems, we recommend using both — you get maximum coverage of grounding issues.

```python
@fe.hallucination(threshold=0.85)
@fe.faithfulness(threshold=0.8)
def test_rag_quality():
    docs = retriever.search(query)
    response = rag_agent(query, context=docs)
    fe.score(response, context=docs)
```

## Best Practices

### Always Provide Context When Available

The metric works best with context to verify against:

```python
# Good: Context provided for verification
@fe.hallucination(threshold=0.85)
def test_with_context():
    context = "Company founded in 2020. 50 employees. Headquarters in Boston."
    response = my_llm("Tell me about the company", context=context)
    fe.score(response, context=context)

# Less effective: No context means relying on general knowledge
@fe.hallucination(threshold=0.85)
def test_without_context():
    response = my_llm("Tell me about company ABC")
    fe.score(response)  # Limited verification possible
```

### Pick the Right Threshold

Creative and conversational apps can work at 0.7-0.8 since some creative license is fine. General knowledge Q&A should target 0.8-0.85. RAG systems need 0.85-0.9 to stay grounded in sources. And for medical, legal, or financial content, push it to 0.9 or above — there's no room for fabrication.

### Track Hallucination Patterns

A common mistake is treating hallucination failures as one-offs. Track them over time — you'll often find the same types of claims getting fabricated repeatedly:

```python
@fe.hallucination(threshold=0.85)
def test_and_analyze():
    response = my_agent(query, context=context)
    result = fe.score(response, context=context)

    if not result.passed:
        # Log for pattern analysis
        log_hallucination_failure(
            query=query,
            response=response,
            context=context,
            reasoning=result.metric_results[0].reasoning
        )
```
