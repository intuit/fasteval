# Completeness Metric

Half-answers frustrate users. The completeness metric checks whether a response actually addresses every part of a query, or if it quietly skips over things the user explicitly asked about.

## Quick Example

```python
import fasteval as fe

@fe.completeness(threshold=0.8)
def test_comprehensive_answer():
    query = "What are the pros and cons of remote work?"
    response = my_llm(query)
    fe.score(response, input=query)
```

## Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `threshold` | float | 0.8 | Minimum score to pass (0.0 - 1.0) |
| `model` | str | None | Override the default LLM model |

## How Scoring Works

The LLM judge evaluates coverage of the query:

| Score | Interpretation | Example |
|-------|---------------|---------|
| **0.9-1.0** | Fully complete | All aspects thoroughly covered |
| **0.7-0.9** | Mostly complete | Minor gaps but main points covered |
| **0.5-0.7** | Partially complete | Some aspects missing |
| **0.0-0.5** | Incomplete | Major gaps in coverage |

### What Gets Evaluated

The judge checks whether every part of the query got addressed, flags obvious information gaps, and verifies that multi-aspect questions didn't get a single-aspect answer. It also considers whether important background context was included and whether the user would need to ask follow-up questions to get what they actually wanted.

## Practical Examples

### Multi-Part Questions

```python
@fe.completeness(threshold=0.85)
def test_comparison():
    query = "Compare Python and JavaScript: syntax, use cases, and performance"
    response = my_llm(query)
    fe.score(response, input=query)
```

### With Expected Content

You can optionally specify expected topics:

```python
@fe.completeness(threshold=0.8)
def test_coverage():
    query = "Explain the water cycle"
    expected_topics = "evaporation, condensation, precipitation, collection"
    response = my_llm(query)
    fe.score(response, expected_topics, input=query)
```

### Checklist-Style Requirements

```python
@fe.completeness(threshold=0.9)
def test_api_documentation():
    query = "Document this API endpoint including: URL, method, parameters, response format, and error codes"
    response = my_llm(f"{query}\n\nEndpoint: POST /users")
    fe.score(response, input=query)
```

## Example Output

**Passing test:**

```

  ✓ completeness: 0.92 (threshold: 0.80)
    Reasoning: The response thoroughly covers all three requested aspects:
    syntax differences are explained with examples, use cases are listed
    for both languages, and performance characteristics are compared.
```

**Failing test:**

```

Evaluation Results:
  ✗ completeness: 0.55 (threshold: 0.80)
    Reasoning: The response covers syntax well but only briefly mentions
    use cases and completely omits the performance comparison that was
    requested.
```

## Choosing Your Threshold

Quick Q&A can work at 0.6-0.7 — users don't always need exhaustive answers. General explanations should hit 0.7-0.8. Documentation needs 0.8-0.85, and if you're checking compliance against a specific checklist, push it to 0.9 or above.

## When to Use Completeness

This metric works best for multi-part questions, documentation generation, comparison tasks, and educational explanations — any query with explicit requirements. If you're after accuracy verification, use `@fe.correctness()` instead. Need brevity? Balance completeness with `@fe.conciseness()`. For format compliance, `@fe.instruction_following()` is your tool.

## Combining with Other Metrics

Balance completeness with other quality aspects:

```python
@fe.completeness(threshold=0.8)   # Does it cover everything?
@fe.conciseness(threshold=0.7)    # Is it efficient?
@fe.correctness(threshold=0.8)    # Is it accurate?
def test_balanced_response():
    query = "Explain the benefits and risks of cloud computing"
    response = my_llm(query)
    fe.score(response, input=query)
```

A common mistake is optimizing for completeness alone. You'll get thorough but bloated responses. Pairing with conciseness keeps things tight.
