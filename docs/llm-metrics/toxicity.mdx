# Toxicity Metric

The toxicity metric is a critical safety guardrail that detects harmful, offensive, or inappropriate content in LLM outputs. Before any LLM reaches users, systematic toxicity evaluation helps ensure responsible AI deployment.

## Overview

Toxicity detection has become a standard component of responsible AI systems. Research has shown that LLMs can generate harmful content under certain conditions—whether through adversarial prompts, edge cases, or model limitations. The toxicity metric provides automated detection of such content, enabling:

- **Pre-deployment testing**: Validate safety before release
- **Continuous monitoring**: Catch regressions in production
- **Compliance**: Meet regulatory and policy requirements
- **Risk management**: Reduce potential for user harm

### Score Interpretation

**Important:** Higher scores indicate LESS toxic content.

| Score | Interpretation | Action |
|-------|---------------|--------|
| **0.95-1.0** | Safe for all audiences | No concerns |
| **0.85-0.95** | Generally safe | Minor concerns, may need context review |
| **0.70-0.85** | Borderline content | Requires review before deployment |
| **0.50-0.70** | Potentially harmful | Should not reach users |
| **0.00-0.50** | Contains toxic content | Immediate attention required |

## What It Detects

The toxicity metric evaluates content across multiple harm categories:

### Primary Categories

| Category | Description | Examples |
|----------|-------------|----------|
| **Hate Speech** | Content targeting identity groups | Slurs, dehumanization, group-based attacks |
| **Harassment** | Content targeting individuals | Bullying, threats, intimidation |
| **Violence** | Threats or graphic violent content | Threats, glorification of violence |
| **Self-Harm** | Dangerous self-directed suggestions | Promotion of self-injury |
| **Sexual Content** | Inappropriate sexual material | Explicit content, objectification |
| **Profanity** | Explicit or vulgar language | Strong language, crude content |

### Contextual Considerations

The metric considers context when evaluating content:

- **Educational content**: Discussions of harmful topics in educational contexts may score higher
- **Quoted content**: Direct quotes may be treated differently than generated content
- **Professional domains**: Medical, legal, or security discussions may contain sensitive terms appropriately

## Parameters

```python
@fe.toxicity(
    threshold=0.95,   # Minimum score to pass (higher = safer required)
    model="gpt-4o",   # Override the default LLM model
)
```

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `threshold` | float | 0.95 | Minimum score to pass (higher scores = less toxic) |
| `model` | str | None | Override the default LLM model for this metric |

## Practical Examples

### User-Facing Chatbot

Every response to users should be evaluated for safety:

```python
import fasteval as fe

@fe.toxicity(threshold=0.95)
def test_customer_response():
    # Responses must be safe even with hostile input
    hostile_input = "This is the worst product ever!"
    response = my_support_bot(hostile_input)
    fe.score(response)
```

### Adversarial Testing

Test with prompts designed to elicit harmful responses:

```python
@fe.toxicity(threshold=0.95)
@fe.csv("adversarial_prompts.csv")
def test_adversarial_safety(prompt):
    """Verify agent doesn't produce toxic output with adversarial inputs."""
    response = my_agent(prompt)
    fe.score(response)
```

### Comprehensive Safety Suite

Combine toxicity with other safety metrics:

```python
@fe.toxicity(threshold=0.95)       # No harmful content
@fe.bias(threshold=0.9)            # No unfair bias
@fe.relevance(threshold=0.7)       # Actually helpful
def test_safe_and_helpful():
    response = my_agent(user_query)
    fe.score(response, input=user_query)
```

### Production Monitoring

Sample and evaluate production traffic:

```python
from fasteval_observe import observe
from fasteval_observe.sampling import FixedRateSamplingStrategy

@observe(
    sampling=FixedRateSamplingStrategy(rate=0.1),  # 10% of traffic
    run_evaluations=True
)
@fe.toxicity(threshold=0.95)
async def production_agent(query: str) -> str:
    return await llm.invoke(query)
```

## Safety Testing Strategy

### Threshold Selection by Application

| Application Type | Recommended Threshold | Rationale |
|-----------------|----------------------|-----------|
| Children's applications | 0.98+ | Maximum safety for vulnerable users |
| General consumer apps | 0.95 | High safety standard |
| Enterprise internal tools | 0.90 | Professional context allows some flexibility |
| Research/development | 0.85 | Controlled environment |

### Layered Safety Approach

Production-ready safety combines multiple layers:

```python
@fe.toxicity(threshold=0.95)       # No harmful language
@fe.bias(threshold=0.9)            # Fair and neutral
@fe.contains()                      # Required disclaimers present
def test_production_safety():
    for prompt in SAFETY_TEST_SUITE:
        response = my_agent(prompt)
        fe.score(response, "professional and helpful")
```

### Building a Safety Test Suite

Include diverse test categories:

1. **Baseline responses**: Normal queries that should pass easily
2. **Edge cases**: Unusual inputs that might trigger unexpected behavior
3. **Adversarial prompts**: Inputs designed to elicit harmful responses
4. **Domain-specific cases**: Industry-relevant safety scenarios

## Best Practices

### Always Test User-Facing Applications

Any LLM that interacts with users should have toxicity evaluation:

```python
@fe.toxicity(threshold=0.95)
def test_all_response_types():
    test_cases = [
        "normal question",
        "angry complaint",
        "edge case request",
    ]
    for query in test_cases:
        response = my_agent(query)
        fe.score(response)
```

### Handle False Positives Appropriately

Some legitimate content may receive lower scores. For domain-specific applications:

1. **Test with domain examples**: Validate against your specific use cases
2. **Adjust thresholds carefully**: Lower only when necessary and documented
3. **Review borderline cases**: Manual review for scores near the threshold
4. **Document exceptions**: Record why certain thresholds were chosen

### Combine with Content Filtering

Toxicity evaluation complements but does not replace content filtering:

- **Pre-generation filtering**: Block known harmful inputs
- **Post-generation evaluation**: Validate outputs with toxicity metric
- **Logging and monitoring**: Track failed evaluations for analysis

### Regular Red Team Testing

Periodically test with new adversarial techniques:

```python
@fe.toxicity(threshold=0.95)
def test_red_team_scenarios():
    """Run periodic red team tests with updated adversarial prompts."""
    for scenario in RED_TEAM_SUITE:
        response = my_agent(scenario.prompt)
        result = fe.score(response)
        if not result.passed:
            log_safety_incident(scenario, result)
```
