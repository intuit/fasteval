# Sampling Strategies [Open in Google Colab](https://colab.research.google.com/github/intuit/fasteval/blob/main/docs/notebooks/langfuse-trace-evaluation/sampling-strategies.ipynb)

Evaluating 100% of production traffic is expensive and slow. **Sampling** gets you statistically meaningful quality signals at a fraction of the cost.

## Why Sample?

Every metric evaluation costs one LLM judge call. That adds up fast:

| Daily Traces | Metrics | Judge Calls/Day | GPT-4o-mini Cost | GPT-4o Cost |
|--------------|---------|-----------------|------------------|-------------|
| 1,000 | 1 | 1,000 | ~$0.30 | ~$5 |
| 1,000 | 3 | 3,000 | ~$0.90 | ~$15 |
| 10,000 | 3 | 30,000 | ~$9 | ~$150 |
| 10,000 | 5 | 50,000 | ~$15 | ~$250 |

*Approximate. Actual cost depends on trace length and metric complexity.*

**Sampling at 2%** of 10k traces with 3 metrics = 600 calls instead of 30,000. That's a 98% cost reduction while still producing statistically meaningful quality signals.

### When NOT to Sample

Don't sample if you have fewer than 100 traces/day — just evaluate everything. Same goes for post-incident audits where you need 100% coverage of the incident window. And dataset evaluations are already curated, so always evaluate all items there.

All strategies are imported from `fasteval_langfuse.sampling`:

```python
from fasteval_langfuse.sampling import (
    NoSamplingStrategy,

)
```

## Random Sampling

**Best for:** General monitoring — an unbiased view of overall quality.

Randomly selects `sample_size` traces from the fetched set. Use `seed` for reproducible runs.

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `sample_size` | `int` | Required | Number of traces to evaluate |
| `seed` | `int` or `None` | `None` | Random seed for reproducibility |

```python
@fe.correctness(threshold=0.8)
@fe.hallucination(threshold=0.9)
@langfuse_traces(
    project="production",
    filter_tags=["customer-support"],
    time_range="last_24h",
    sampling=RandomSamplingStrategy(sample_size=200, seed=42),
)
def test_production_quality(trace_id, input, output, context, metadata):
    """Evaluate 200 random traces from the last 24 hours."""
    expected = metadata.get("expected_answer")
    fe.score(output, expected, context=context, input=input)
```

**Tip:** Use `seed=42` (or any fixed value) in CI for deterministic runs. Omit `seed` for true randomness in nightly evaluations.

## Stratified Sampling

**Best for:** Ensuring fairness across user segments — e.g., free vs. paid, different regions.

Groups traces by a metadata key and samples evenly from each group. This prevents the most active segment from dominating your eval results.

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `strata_key` | `str` | Required | Metadata key to group by (supports dot notation: `metadata.user_type`) |
| `samples_per_stratum` | `int` | Required | Number of traces to sample per group |

```python
@fe.correctness(threshold=0.8)
@fe.relevance(threshold=0.75)
@langfuse_traces(
    project="production",
    filter_tags=["customer-query"],
    time_range="last_7d",
    sampling=StratifiedSamplingStrategy(
        strata_key="metadata.user_type",  # Groups: "free", "pro", "enterprise"
        samples_per_stratum=30,
    ),
)
def test_quality_across_user_types(trace_id, input, output, context, metadata):
    """Sample evenly across free/paid users to ensure fair quality."""
    user_type = metadata.get("user_type", "unknown")
    fe.score(output, input=input)
```

If a group has fewer traces than `samples_per_stratum`, all traces from that group are included.

## Score-Based Sampling

**Best for:** Failure analysis — spend your evaluation budget where it matters most.

This strategy oversamples traces with low existing scores (e.g., user ratings) and undersamples high-scoring ones. In practice, it's the most cost-effective way to find and understand failures.

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `score_name` | `str` | Required | Langfuse score name to sample by (e.g., `user_rating`) |
| `low_score_threshold` | `float` | Required | Scores below this are "low" |
| `low_score_rate` | `float` | Required | Fraction of low-score traces to include (0.0 - 1.0) |
| `high_score_rate` | `float` | Required | Fraction of high-score traces to include (0.0 - 1.0) |

```python
@fe.relevance(threshold=0.7)
@fe.faithfulness(threshold=0.8)
@langfuse_traces(
    project="production",
    filter_tags=["rag"],
    time_range="last_7d",
    sampling=ScoreBasedSamplingStrategy(
        score_name="user_rating",
        low_score_threshold=3.0,
        low_score_rate=1.0,   # Evaluate 100% of low-rated traces
        high_score_rate=0.05, # Evaluate only 5% of high-rated traces
    ),
)
def test_low_rated_interactions(trace_id, input, output, context, metadata):
    """Focus evaluation budget on traces with poor user ratings."""
    if not context:
        context = metadata.get("docs") or metadata.get("retrieval_context")
    expected = metadata.get("expected_answer")
    fe.score(output, expected, context=context, input=input)
```

**Tip:** Set `high_score_rate=0.0` to evaluate *only* failures — the cheapest possible configuration.

## Recent-First Sampling

**Best for:** Post-deployment monitoring — catch regressions as fast as possible.

Uses exponential decay so newer traces are much more likely to be selected than older ones.

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `decay_factor` | `float` | Required | Decay rate (0.0 - 1.0). Higher = stronger recency bias. |

```python
@fe.correctness(threshold=0.8)
@langfuse_traces(
    project="production",
    time_range="last_24h",
    limit=100,
    sampling=RecentFirstSamplingStrategy(decay_factor=0.8),
)
def test_recent_deployment(trace_id, input, output, metadata):
    """Prioritize evaluating the most recent traces."""
    fe.score(output, input=input)
```

## No Sampling

**Best for:** Small projects, post-incident audits, or full coverage requirements.

Evaluates every fetched trace. This is the **default** when no `sampling` parameter is specified.

```python
@fe.correctness(threshold=0.8)
@fe.hallucination(threshold=0.9)
@langfuse_traces(
    time_range="2026-03-04 14:00 to 2026-03-04 16:00",
    sampling=NoSamplingStrategy(),
)
def test_incident_audit(trace_id, input, output, context, metadata):
    """Evaluate every trace during the incident window."""
    fe.score(output, context=context, input=input)
```

**Warning:** Always combine with `limit` or a narrow `time_range` to avoid evaluating thousands of traces by accident.

## Scheduling Patterns

Different strategies fit different schedules.

### Pattern A: Hourly Quick Check

**Goal:** Is the system healthy right now? Use `RandomSamplingStrategy(sample_size=20)` with 1-2 core metrics over `last_1h`. That's roughly 20 judge calls per hour.

```python
@fe.correctness(threshold=0.8)
@langfuse_traces(
    time_range="last_1h",
    sampling=RandomSamplingStrategy(sample_size=20),
)
def test_hourly_health(trace_id, input, output, metadata):
    fe.score(output, input=input)
```

### Pattern B: Nightly Deep Dive

**Goal:** Full quality report across all dimensions. Use `ScoreBasedSamplingStrategy` or `StratifiedSamplingStrategy` with 3-5 metrics over `last_24h`. Expect 500-1000 judge calls per night.

```python
@fe.correctness(threshold=0.8)
@fe.relevance(threshold=0.8)
@fe.faithfulness(threshold=0.9)
@fe.hallucination(threshold=0.9)
@langfuse_traces(
    time_range="last_24h",
    sampling=ScoreBasedSamplingStrategy(
        score_name="user_rating",
        low_score_threshold=3.0,
        low_score_rate=1.0,
        high_score_rate=0.1,
    ),
)
def test_nightly_deep_dive(trace_id, input, output, context, metadata):
    fe.score(output, context=context, input=input)
```

### Pattern C: Post-Incident Audit

**Goal:** What went wrong during the incident? Use `NoSamplingStrategy` with a full metric stack over a specific absolute time window.

```python
@fe.correctness(threshold=0.7)
@fe.hallucination(threshold=0.9)
@langfuse_traces(
    time_range="2026-03-04 14:00 to 2026-03-04 16:00",
    sampling=NoSamplingStrategy(),
)
def test_incident_audit(trace_id, input, output, context, metadata):
    fe.score(output, context=context, input=input)
```

## GitHub Actions Automation

Automate your evaluation schedule with cron-triggered workflows:

```yaml
# .github/workflows/nightly-eval.yml
name: Nightly Production Evaluation

on:
  schedule:
    - cron: '0 8 * * *'   # 8 AM UTC daily
  workflow_dispatch:        # Manual trigger for ad-hoc runs

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: pip install fasteval-core fasteval-langfuse pytest

      - name: Run nightly evaluation
        env:
          LANGFUSE_PUBLIC_KEY: ${{ secrets.LANGFUSE_PUBLIC_KEY }}
          LANGFUSE_SECRET_KEY: ${{ secrets.LANGFUSE_SECRET_KEY }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: pytest tests/test_nightly_eval.py -v --tb=short
```

A few things we've learned from running these in practice: always set `limit` or `sample_size` to prevent surprise bills if traffic spikes overnight. Use `seed` in random sampling for reproducible CI runs you can debug later. And keep separate test files by schedule — `test_hourly.py`, `test_nightly.py`, `test_regression.py`.

## Cost Optimization

| Lever | Impact | Example |
|-------|--------|---------|
| **Reduce sample size** | Linear | 200 to 50 traces = 75% savings |
| **Fewer metrics** | Linear | 5 to 2 metrics = 60% savings |
| **Cheaper judge model** | ~10x | GPT-4o to GPT-4o-mini |
| **Score-based sampling** | ~20x | 100% failures + 5% successes |
| **Hourly/nightly split** | ~5x | Lightweight hourly, deep nightly |

We recommend starting with `RandomSamplingStrategy(sample_size=50)` and 1-2 metrics. Add more as you understand your quality patterns and failure modes.
