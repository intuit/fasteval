# Tool Sequence

Order matters. A payment agent that charges a card before validating it isn't just wrong -- it's dangerous. `tool_sequence` checks whether your agent called tools in the right order. It's a **deterministic metric**, no LLM calls needed.

## Usage

```python
import fasteval as fe
from fasteval import ExpectedTool

@fe.tool_sequence(threshold=0.8)
def test_workflow():
    response, tool_calls = agent("Process the order")

    expected = [
        ExpectedTool(name="validate_order"),
        ExpectedTool(name="charge_payment"),
        ExpectedTool(name="send_confirmation"),
    ]

    fe.score(response, tool_calls=tool_calls, expected_tools=expected)
```

## Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `threshold` | float | 0.8 | Minimum score to pass (0.0 - 1.0) |
| `strict` | bool | False | Require exact sequence match |
| `weight` | float | 1.0 | Weight for score aggregation |

## How It Works

![Tool Sequence LCS Matching](docs/assets/tool-sequence-lcs.svg)

### Non-Strict Mode (Default)

Uses the **Longest Common Subsequence (LCS)** algorithm:

```
score = LCS_length / max(expected_length, actual_length)
```

This gives partial credit when extra tools appear between expected ones, or when some tools are out of order.

### Strict Mode

All or nothing. The sequences must match exactly -- score is 1.0 if they do, 0.0 if they don't.

```python
@fe.tool_sequence(strict=True)  # All or nothing
```

## Examples

### Non-Strict Matching

```python
# Expected: A → B → C
# Actual:   A → X → B → C
# LCS: A, B, C (length 3)
# Score: 3/4 = 0.75 ✓ (extra X allowed)

# Expected: A → B → C
# Actual:   A → C → B
# LCS: A, C (length 2) or A, B (length 2)
# Score: 2/3 = 0.67 (partial order match)
```

### Strict Matching

```python
# Expected: A → B → C
# Actual:   A → B → C
# Score: 1.0 ✓

# Expected: A → B → C
# Actual:   A → X → B → C
# Score: 0.0 ✗ (extra tool breaks strict match)
```

## Example Output

```
tool_sequence: 0.75 (threshold: 0.80) ✗
Reasoning: Sequence similarity: 3/4 tools in order
Details:
  - Expected: validate → charge → confirm
  - Actual: validate → log → charge → confirm
  - Matched: validate, charge, confirm (in order)
```

## Use Cases

### Payment Processing Workflow

```python
@fe.tool_sequence(threshold=1.0, strict=True)  # Must be exact
def test_payment_flow():
    # Payment must follow exact order for compliance
    expected = [
        ExpectedTool(name="validate_card"),
        ExpectedTool(name="check_fraud"),
        ExpectedTool(name="charge_card"),
        ExpectedTool(name="send_receipt"),
    ]
```

### Flexible Research Workflow

```python
@fe.tool_sequence(threshold=0.6, strict=False)  # Allow flexibility
def test_research_agent():
    # Research can be done in various orders
    expected = [
        ExpectedTool(name="search_*"),
        ExpectedTool(name="summarize"),
    ]
```

## Combining with Other Metrics

```python
@fe.tool_call_accuracy(threshold=0.8)  # Were correct tools called?
@fe.tool_sequence(threshold=0.8)        # In correct order?
@fe.tool_args_match(threshold=0.9)      # With correct arguments?
def test_complete_workflow():
    response, tool_calls = agent(task)
    expected = [...]
    fe.score(response, tool_calls=tool_calls, expected_tools=expected)
```
