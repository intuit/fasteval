# Metrics Overview

Everything in fasteval revolves around metrics. You pick the ones that matter for your use case, set thresholds, and let them score your LLM's output on a 0.0 to 1.0 scale — with reasoning attached so you can actually understand *why* something scored the way it did.

We've organized metrics into four categories. Some are fast and deterministic, others use an LLM judge under the hood. The tradeoff is always speed vs. depth of understanding.

![Metric Categories](docs/assets/metrics-categories.svg)

## Metric Categories

| Category | Evaluation Method | Speed | Primary Use Cases |
|----------|------------------|-------|-------------------|
| **LLM-Based Metrics** | LLM judge assessment | ~1-3s | Semantic understanding, quality assessment |
| **RAG Metrics** | LLM judge with context | ~1-3s | Retrieval pipeline evaluation |
| **Deterministic Metrics** | Rule-based computation | Instant | Format validation, CI/CD pipelines |
| **Conversation Metrics** | Multi-turn LLM analysis | ~2-5s | Chatbot and dialogue systems |

### When to Use Each Category

**LLM-Based Metrics** are your go-to when surface-level text matching won't cut it. You need them for evaluating correctness, relevance, safety, and any quality dimension that requires understanding what the text actually *means* — not just what characters it contains.

**RAG Metrics** come into play when you're building retrieval-augmented generation pipelines. They're specifically designed to evaluate the relationship between what your model generated and what the source documents actually say. If your answers need to be grounded in retrieved context, these are non-negotiable.

**Deterministic Metrics** are instant and don't hit any API. Use them for exact matching, format validation, and anywhere you'd rather not wait on an LLM call. They're perfect for CI/CD pipelines, and honestly, they're often all you need for structural checks. Don't overthink it.

**Conversation Metrics** matter when you're building multi-turn systems — chatbots, support agents, assistants. They look at the full conversation history to evaluate context retention, consistency, and whether the bot stays on topic across turns.

## LLM-Based Metrics

LLM-based metrics use a judge model to evaluate the *meaning* of responses. They get paraphrasing. They understand nuance. And they're the only way to catch subtle quality issues that rule-based checks will miss entirely.

### Available Metrics

**Correctness** measures semantic accuracy against an expected output. It's the workhorse metric for Q&A systems and factual queries — did the model get it right?

**Relevance** checks whether the response actually addresses the question that was asked. Useful across pretty much all conversational AI.

**Hallucination** flags content that isn't grounded in facts or provided context. Essential for knowledge-based systems where making things up is a dealbreaker.

**Toxicity** evaluates whether content is safe and appropriate. If your app is user-facing, you want this one.

**Bias** assesses fairness and neutrality. In practice, it's critical for sensitive domains — hiring tools, financial advice, anything where slanted output causes real harm.

**Coherence** looks at logical consistency within the response. Long-form generation tends to drift, and this catches it.

**Conciseness** judges whether the response is appropriately brief. Summarization tasks and customer support bots benefit most here.

### How They Work

LLM-based metrics construct evaluation prompts that ask the judge model to:
1. Analyze the actual output against the criteria
2. Consider the expected output (if provided) as reference
3. Account for input context (if provided)
4. Return a score (0.0-1.0) with detailed reasoning

### Example

```python
import fasteval as fe

@fe.correctness(threshold=0.8)
@fe.relevance(threshold=0.7)
def test_qa_bot():
    response = my_bot("What year was Python released?")
    fe.score(response, "Python was first released in 1991", input="What year was Python released?")
```

### Threshold Guidelines

For **Correctness**, we recommend 0.75–0.85. The model needs to be substantially accurate — but perfection in wording isn't the goal. **Relevance** sits comfortably at 0.70–0.80; the response should address the question, though tangential context is sometimes fine. **Hallucination** thresholds should be higher — 0.80–0.90 — because ungrounded content erodes trust fast.

**Toxicity** demands the strictest range: 0.90–0.95. You don't want much tolerance for harmful output. **Bias** works well at 0.80–0.90, giving moderate room while still catching significant skew. **Coherence** at 0.70–0.80 is reasonable since minor logical hiccups aren't always critical. And **Conciseness** is the most lenient at 0.60–0.75 — it's a style metric, so we give it more slack.

## RAG Metrics

RAG metrics evaluate retrieval-augmented generation pipelines end to end. They look at both the quality of what you retrieved *and* whether the generated answer stayed faithful to those documents.

### Available Metrics

**Faithfulness** is the big one. It checks whether the response is grounded in retrieved documents — catching those cases where the model generates something plausible but completely unsupported by the context. If you're running a RAG system, this is the metric you start with.

**Contextual Precision** measures your retriever's signal-to-noise ratio. Are the documents it's returning actually relevant to the query? Low precision means your retriever is pulling in junk.

**Contextual Recall** is the flip side — do the retrieved documents contain the information needed to answer the query? Low recall means your knowledge base has gaps, or your retriever isn't finding what's there.

**Answer Correctness** gives you a combined RAG quality score. It's the single-number summary when you want an overall read on how well the whole pipeline is working.

### Example

```python
import fasteval as fe

@fe.faithfulness(threshold=0.8)
@fe.contextual_precision(threshold=0.7)
def test_rag_pipeline():
    docs = retriever.search("company refund policy")
    response = rag_agent("How do I get a refund?", context=docs)
    fe.score(response, context=docs, input="How do I get a refund?")
```

### Threshold Guidelines

**Faithfulness** should sit at 0.80–0.90. Grounding matters — a lot. A common mistake is setting this too low and then wondering why users complain about made-up answers. **Contextual Precision** and **Contextual Recall** both work well at 0.70–0.80. Some noise in retrieval results is normal, and complete coverage isn't always realistic. **Answer Correctness** at 0.75–0.85 gives you a solid overall quality bar without being punishingly strict.

## Deterministic Metrics

No LLM calls. No latency. No cost. Deterministic metrics use rule-based computation for exact matching, pattern validation, and format checking. They run instantly.

### Available Metrics

**Contains** checks whether a substring exists in the output. Think required keywords, legal disclaimers, mandatory phrases — anything where specific text *must* appear.

**Exact Match** verifies that strings are identical. It's strict by design. Use it for structured outputs where the format has to be precisely right.

**ROUGE** measures text overlap similarity between the output and a reference. What we've found is it works best for summarization quality, where you want to confirm the key information survived.

**JSON Schema** validates that output is valid JSON matching a given schema. If your LLM is producing API responses or structured data, this is how you catch malformed output before it hits production.

**Regex** does pattern matching. Format validation, phone numbers, dates, IDs — any case where the shape of the text matters more than the meaning.

### When to Use Deterministic Metrics

In CI/CD pipelines, they're a no-brainer — fast execution with zero API calls. For format validation and schema compliance, they're the right tool. When you need to verify required content like legal disclaimers or mandatory phrases, **Contains** is simpler and cheaper than spinning up a judge model. For high-volume test suites where cost matters, deterministic metrics keep your bill under control. And they pair well with LLM metrics — layer them together so you get both structural checks and semantic evaluation.

### Example

```python
import fasteval as fe

@fe.contains()
def test_includes_disclaimer():
    response = legal_bot("What are your terms?")
    fe.score(response, "not legal advice")  # Must contain this phrase

@fe.json_schema(schema={"type": "object", "required": ["id", "status"]})
def test_api_format():
    response = my_api("get status")
    fe.score(response, None)  # Schema validation only
```

## Conversation Metrics

Conversation metrics evaluate multi-turn dialogues. They analyze how well a system holds context, stays consistent, and keeps the conversation on track across multiple exchanges.

### Available Metrics

**Context Retention** evaluates whether the bot remembers information from earlier turns. A support bot should recall that order number you mentioned three messages ago. When it doesn't, users notice immediately.

**Consistency** catches contradictions across turns. If the bot says delivery takes 3 days in one response and 5 days in the next, that's a consistency failure. The gotcha here is that inconsistencies often only surface in longer conversations — so test with realistic turn counts.

**Topic Drift** monitors whether responses stay relevant to the conversation's goal. It detects when the system wanders off into unrelated territory. Focused task assistants — booking bots, troubleshooting agents — need this most.

### Example

```python
import fasteval as fe

@fe.context_retention(threshold=0.8)
@fe.consistency(threshold=0.9)
@fe.conversation([
    {"query": "My order is #12345", "expected": "order 12345"},
    {"query": "When will it arrive?", "expected": "delivery"},  # Should remember order
])
def test_support_conversation(query, expected, history):
    response = my_bot(query, history=history)
    fe.score(response, expected, input=query)
```

## Choosing the Right Metrics

### For General Q&A and Chatbots

Start with core quality and safety metrics:

```python
@fe.correctness(threshold=0.8)  # Factual accuracy
@fe.relevance(threshold=0.7)    # Answers the question
@fe.toxicity(threshold=0.95)    # Safe content
```

### For RAG Systems

Focus on retrieval quality and grounding:

```python
@fe.faithfulness(threshold=0.8)       # Grounded in context
@fe.answer_correctness(threshold=0.7) # Overall quality
@fe.hallucination(threshold=0.85)     # No fabricated content
```

### For Production Safety

Prioritize safety metrics with strict thresholds:

```python
@fe.toxicity(threshold=0.95)    # No harmful content
@fe.bias(threshold=0.9)         # Fair and neutral
@fe.contains()                   # Required disclaimers
```

### For Fast CI/CD

Use deterministic metrics to avoid API latency:

```python
@fe.contains()                         # Required content
@fe.exact_match()                      # Specific format
@fe.json_schema(schema=my_schema)      # Valid structure
```

### Combining Metrics for Full Evaluation

Production evaluation usually means mixing metric types:

```python
@fe.correctness(threshold=0.8)     # LLM: semantic accuracy
@fe.faithfulness(threshold=0.8)    # RAG: grounded in context
@fe.contains()                      # Fast: required phrase
@fe.json_schema(schema=my_schema)  # Fast: valid structure
def test_comprehensive():
    ...
```

All metrics run and report independently. The test fails if **any** metric falls below its threshold — so you'll know exactly which quality dimension broke and why.
