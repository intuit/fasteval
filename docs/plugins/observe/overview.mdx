# fasteval-observe

Runtime monitoring for LLM applications in production. Sample traffic, run evaluations async, and log structured results -- all without blocking your requests.

## What is fasteval-observe?

**fasteval-observe** is an optional plugin that takes fasteval beyond test-time evaluation into production monitoring. Core fasteval handles CI/CD and development testing. fasteval-observe picks up where that leaves off: it samples live traffic and runs evaluations asynchronously against real user interactions.

```bash
# Install separately from fasteval
pip install fasteval-core
pip install fasteval-observe
```

```python
import fasteval as fe
from fasteval_observe import observe
from fasteval_observe.sampling import FixedRateSamplingStrategy

# Monitor 5% of production agent calls
@observe(sampling=FixedRateSamplingStrategy(rate=0.05), run_evaluations=True)
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
async def my_agent(query: str) -> str:
    response = await llm.invoke(query)
    return response
```

## The Problem It Solves

Synthetic test cases can't anticipate every production scenario. Real users ask questions, phrase requests, and hit edge cases that test suites miss. Your unit tests cover known scenarios but can't predict all user inputs. Integration tests rely on synthetic data. Staging sees different traffic patterns than production. And in production itself, there's typically no systematic quality evaluation happening.

fasteval-observe bridges that gap by sampling real production traffic and running the same fasteval metrics you already use in testing. It detects regressions that synthetic tests miss, surfaces edge cases from actual user behavior, tracks quality trends over time, and alerts on degradation before users notice.

## Key Features

| Feature | Description |
|---------|-------------|
| **Async Evaluation** | Non-blocking processing adds less than 1ms to request latency |
| **Pluggable Sampling** | 6 built-in strategies + custom strategy support |
| **Structured Logging** | JSON format for Kubernetes sidecars, Fluentd, Datadog, etc. |
| **Circuit Breaker** | Automatic protection against queue overflow under load |
| **fasteval Integration** | Run any fasteval metric on sampled production calls |
| **Context Extraction** | Extract RAG context for faithfulness evaluation |

Six sampling strategies ship out of the box: FixedRate for steady-state monitoring, Adaptive for incident detection, TokenBudget for cost control, TimeWindow for rate-limited APIs, Priority for critical path focus, and Composite for combining multiple strategies.

## Architecture

fasteval-observe uses a fire-and-forget architecture to keep latency impact near zero.

Here's the request flow: the agent receives a request, a sub-millisecond sampling check runs, the function executes normally, and the response returns to the user. If the call was sampled, the observation gets queued asynchronously (non-blocking). A background worker thread then batches and logs the results.

| Operation | Latency Impact |
|-----------|---------------|
| Sampling decision | < 0.01ms |
| Async queue enqueue | < 0.1ms |
| Total request overhead | < 1ms |
| Background evaluation | Zero (async) |

```
Request → Agent Function → Response
              ↓ (if sampled)
        Async Queue (non-blocking)
              ↓
     Background Worker Thread
              ↓
    Structured JSON Logs
              ↓
    External Collectors
    (Prometheus, Datadog, etc.)
```

## When to Use

**Use fasteval-observe for** production monitoring of real user interactions, quality trending over time, incident investigation with increased sampling, cost tracking of token usage, and compliance-driven quality assessment.

**Keep using core fasteval for** pytest-based evaluation in CI/CD, local development with immediate feedback, offline batch evaluation of datasets, and interactive human review workflows.

### Complementary Usage

Most teams run both. We recommend it:

```python
# Development and CI/CD: Core fasteval
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
def test_my_agent():
    response = my_agent("test query")
    fe.score(response, expected)

# Production: fasteval-observe wrapping the same metrics
@observe(sampling=AdaptiveSamplingStrategy(...), run_evaluations=True)
@fe.correctness(threshold=0.8)
@fe.toxicity(threshold=0.95)
async def production_agent(query: str) -> str:
    return await llm.invoke(query)
```

## Package Separation

fasteval-observe is architecturally separate from core fasteval. The coupling is one-way: fasteval-observe imports from fasteval, but core fasteval has zero knowledge of the observe plugin. They're installed independently and versioned independently.

This keeps core fasteval lightweight for testing, prevents production dependencies from leaking into test environments, lets teams adopt monitoring incrementally, and ensures breaking changes in one package don't affect the other.
