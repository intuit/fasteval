# Score Reporting

## Understanding Score Reporting

When you evaluate traces, the plugin pushes quality scores back to Langfuse and attaches them to the original traces. This means your quality data and operational data live in the same place.

### Why This Matters

Without score reporting, evaluation results only live in pytest output — useful for CI/CD gates but invisible to the rest of the team. With it, quality metrics show up in Langfuse alongside everything else, accessible to anyone with dashboard access.

Scores persist in Langfuse, so you can track trends over time. When aggregate scores flag a problem, you can filter down to the exact low-scoring traces and dig in. And since Langfuse supports alerting on score values, you can get notified automatically when quality drops.

### The Reporting Workflow

Score reporting happens as part of trace evaluation. After fasteval metrics assess a trace's quality (producing scores from 0.0 to 1.0), the plugin pushes those scores via Langfuse's API, and they attach to the original trace. This happens per-trace, typically batched for efficiency.

## What Gets Reported

### Individual Metric Scores

Each metric you configure produces a separate score in Langfuse:

```python
@fe.correctness(threshold=0.8)
@fe.relevance(threshold=0.75)
@fe.hallucination(threshold=0.9)
@langfuse_traces(project="production")
def test_quality(trace_id, input, output, context, metadata):
    fe.score(output, input=input)
```

**Resulting scores in Langfuse:**

| Score Name | Value | Description |
|------------|-------|-------------|
| `fasteval_correctness` | 0.87 | Factual accuracy assessment |
| `fasteval_relevance` | 0.82 | Response addresses the query |
| `fasteval_hallucination` | 0.94 | Absence of fabricated content |
| `fasteval_aggregate` | 0.877 | Combined score across all metrics |

### Aggregate Score

An aggregate score is always reported alongside individual metrics. It's typically the average of all metric scores and gives you a single number for dashboards and alerting.

### Score Reasoning (Comments)

When the LLM judge provides an explanation, it's stored in the score's comment field:

```json
{
  "name": "fasteval_correctness",
  "value": 0.87,
  "comment": "The response accurately describes the password reset process. However, it omits the optional two-factor authentication step, which prevents a perfect score.",
  "source": "fasteval"
}
```

This reasoning shows up in Langfuse when viewing individual scores, which helps with root cause analysis.

## Score Names and Prefixes

### Default Naming Convention

All scores get prefixed with `fasteval_` by default: `fasteval_correctness`, `fasteval_relevance`, `fasteval_hallucination`, `fasteval_aggregate`. The prefix provides namespacing (so you can tell fasteval scores from other systems), easy filtering (`score.fasteval_* < 0.7`), and clarity about where the score came from.

### Custom Prefixes

```python
from fasteval_langfuse import configure_langfuse, LangfuseConfig

configure_langfuse(LangfuseConfig(
    score_name_prefix="quality_"
))

# Now scores are named:
# - quality_correctness
# - quality_relevance
# - quality_aggregate
```

You might use `prod_` / `staging_` for environment separation, `v2_` for version tracking, or `nlp_` / `ml_` for team separation.

### No Prefix

For simpler names, drop the prefix entirely:

```python
configure_langfuse(LangfuseConfig(
    score_name_prefix=""
))

# Scores are now simply:
# - correctness
# - relevance
# - aggregate
```

Be careful with this if other systems also write scores to Langfuse — you might get name collisions.

### Filtering by Score

Use Langfuse's filters to find traces by score:

```
Filter: score.fasteval_correctness < 0.7
→ Shows all traces that failed correctness evaluation
```

Common patterns: `score.fasteval_aggregate < 0.7` for low overall quality, `score.fasteval_hallucination < 0.8` for potential hallucinations, `score.fasteval_correctness IS NULL` for unevaluated traces, and `score.fasteval_aggregate > 0.9` for high-quality examples worth promoting to datasets.

### Analytics and Dashboards

Langfuse gives you analytics views for scores. **Score distribution** histograms show your quality spread — a bimodal distribution (peaks at high and low) suggests distinct success/failure modes. **Score trends** track averages over time; gradual declines mean systemic issues, sudden drops point to specific incidents. **Score correlation** lets you compare fasteval scores against user ratings to validate your evaluation approach.

### Setting Up Alerts

To get proactive notifications:

1. Go to **Alerts** in Langfuse
2. Set a condition like `avg(score.fasteval_aggregate) < 0.75`
3. Pick a time window (e.g., last 1 hour)
4. Choose your notification channel (Slack, email, webhook)

## Configuration Options

### Enabling/Disabling Auto-Push

```python
# Global: disable for all tests
configure_langfuse(LangfuseConfig(
    auto_push_scores=False
))

# Per-test: override for specific test
@langfuse_traces(project="prod", auto_push_scores=True)
def test_with_push(trace_id, input, output, context, metadata):
    fe.score(output, input=input)
```

Turn off auto-push for dev/debugging (to avoid polluting data), dry runs, or when you've got custom score handling logic.

### Score Name Prefix

```python
configure_langfuse(LangfuseConfig(
    score_name_prefix="fasteval_"     # Default
    # or
    score_name_prefix="quality_v2_"   # Custom
    # or
    score_name_prefix=""              # No prefix
))
```

### Retry Behavior

```python
configure_langfuse(LangfuseConfig(
    retry_on_failure=True  # Default: retry on transient errors
))
```

Network blips or Langfuse API hiccups can cause occasional push failures. With retries on, the plugin tries again before giving up.

## Operational Considerations

### Score Push Timing

Scores are pushed after each trace evaluation, not batched at the end. This means you get real-time visibility as evaluation progresses, partial results survive if evaluation crashes mid-run, and each trace generates API calls (one per score plus the aggregate).

### Network and API Limits

For large evaluations, keep these in mind. Langfuse has rate limits — the plugin handles this with batching and backoff, but very high volume may cause delays. Network latency also adds up across thousands of traces.

```python
configure_langfuse(LangfuseConfig(
    batch_size=50  # Traces per batch; affects API efficiency
))
```

### Handling Push Failures

If a push fails: with retry enabled, the plugin retries. If that also fails, the error gets logged and evaluation continues. A summary at the end shows any failed pushes. Evaluation results still appear in pytest output regardless.

### Flushing Scores

The plugin flushes pending scores automatically at the end of evaluation. For explicit control:

```python
from fasteval_langfuse import LangfuseClient

client = LangfuseClient()
# ... perform evaluations ...
client.flush()  # Ensure all pending scores are pushed
```

### Idempotency

Pushing the same score twice updates the existing score rather than creating a duplicate. Re-runs are safe.

### Data Volume

Each score adds to your Langfuse storage. For high-volume evaluation, think about evaluation frequency (hourly vs. daily), use sampling to reduce traced evaluated, and review Langfuse's storage/retention policies.

## Use Case Examples

### Quality Monitoring with Alerting

```python
@fe.correctness(threshold=0.8)
@langfuse_traces(
    project="production",
    time_range="last_1h",
    sampling=RandomSamplingStrategy(sample_size=100)
)
def test_hourly_quality(trace_id, input, output, context, metadata):
    fe.score(output, input=input)
```

After this runs, 100 traces have scores in Langfuse. Set an alert: `avg(score.fasteval_aggregate) < 0.75` over the last hour, notify #quality-alerts on Slack.

### Debugging with Score Filters

When investigating quality issues: run evaluation against the relevant time period, filter in Langfuse for `score.fasteval_correctness < 0.6`, review the low-scoring traces, and check the score comments for the judge's reasoning.

### Environment-Specific Scores

Keep environments separate with different prefixes:

```python
import os

env = os.getenv("ENVIRONMENT", "staging")

configure_langfuse(LangfuseConfig(
    score_name_prefix=f"fasteval_{env}_"
))

# Production scores: fasteval_production_correctness
# Staging scores: fasteval_staging_correctness
```
